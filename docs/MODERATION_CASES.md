# Moderation Cases

> The unified moderation system. **Every sanction in Moddy is a case** — there is
> no separate "blacklist table" or per-feature sanction store. Subsystems record
> sanctions through one scalable service; the bot, the internal API and the
> dashboard all read the same three tables.

---

## 1. Concept

A **case** is a moderation folder, decoupled from the Discord context. Three
notions must never be confused:

- **subject** — who/what the case is about (the sanctioned user or guild).
- **issuer** — who created the case.
- **scope** — *where* the case applies (a server, the network, the platform…).

Every actor/target is a `*_type` (enum) + `*_id` (TEXT) couple, so new kinds of
subjects/issuers/scopes can be added without a structural migration.

A case carries **one or more sanctions** (`case_sanctions`, each with its own
lifecycle) and a chronological **event timeline** (`case_events`: comments,
evidence, notes and system events).

### Status lifecycle

A case is `open` while at least one of its sanctions is `active`; it auto-closes
when the last active sanction is revoked or expires. A **manual** open/close by a
moderator sets `status_locked = TRUE`, so sanction lifecycle changes never
auto-flip a case a human set by hand.

Every status transition (auto or manual) writes a `status_change` event with a
`trigger` (`manual` / `expiration` / `revocation` / `system`). The `cases` row
never stores *why* it is closed — the timeline does.

## 2. Schema

Tables (created in `db/base.py::_init_tables`):

- `cases` — the folder (`id` UUID PK, public `reference`, `type`, subject/issuer/
  scope couples, `reason`, `status`, `status_locked`, `group_id`).
- `case_sanctions` — sanctions (`action`, `expires_at`, `status`, `issued_by_*`,
  `revoked_*`).
- `case_events` — the timeline (`type`, `author_*`, `content`, `payload` JSONB).

PostgreSQL ENUM types: `case_type`, `subject_type`, `issuer_type`, `scope_type`,
`case_status`, `sanction_action`, `sanction_status`, `event_type`, `author_type`.
`sanction_action` is extensible (`ALTER TYPE … ADD VALUE`).

The public `reference` is a 6-char code from the unambiguous alphabet
`ABCDEFGHJKMNPQRSTUVWXYZ23456789` (no O/0, I/1), generated with retry on the
`UNIQUE` constraint. It is never derived from the internal UUID.

UUIDs are generated by the bot (Python `uuid4`) — no `pgcrypto`/`gen_random_uuid`
dependency.

## 4. The scalable sanction service

`services/case_service.py` is the **only** thing subsystems call to record a
sanction. They name a *source*; a source describes how that subsystem maps onto
the case model:

```python
@dataclass(frozen=True)
class CaseSource:
    key: str
    case_type: CaseType
    scope_type: ScopeType
    subject_type: SubjectType
    actions: list[SanctionAction]
    manual: bool             # selectable in /mod case create
    requires_scope_id: bool  # e.g. the guild id for "guild"
```

Two sources ship today (`SOURCES` registry):

| key | case_type | scope | created by | actions |
|---|---|---|---|---|
| `global` | `global` | `platform` | Moddy staff (manual) | warn, restrict, ban |
| `guild` | `guild` | the server | auto (Discord events) | warn, mute, kick, ban |

`global` covers **Moddy-team blacklists and global sanctions** (a `ban` here is a
full bot blacklist — `cogs/blacklist_check.py` reads it). `guild` covers
**per-server sanctions**.

### Adding a new sanction kind

One line — no schema change, no new table:

```python
from services.case_service import register_source, CaseSource
from utils.moderation_cases import CaseType, ScopeType, SubjectType, SanctionAction

register_source(CaseSource(
    key="network",
    case_type=CaseType.NETWORK,
    scope_type=ScopeType.NETWORK,
    subject_type=SubjectType.DISCORD_USER,
    actions=[SanctionAction.WARN, SanctionAction.MUTE, SanctionAction.BAN],
    manual=True,
))
```

### Recording / lifting

```python
await bot.cases.record_sanction(
    "guild", subject_id=user_id, action="ban", reason="...",
    issuer_type="discord_user", issuer_id=mod_id, scope_id=guild_id,
)
await bot.cases.revoke_sanction(
    "guild", subject_id=user_id, action="ban", scope_id=guild_id,
    by_type="discord_user", by_id=mod_id,
)
```

`record_sanction` **de-duplicates** by default: a new sanction for a subject who
already has an *open* case of the same `(type, scope)` is appended to that case
instead of opening a second folder (`link_open=True`).

## 5. Auto-recording guild sanctions

`cogs/case_sync.py` listens to `on_audit_log_entry_create` so a ban / kick /
timeout on **any** server Moddy is in opens a case automatically — **even when
the action did not go through Moddy**. The audit log gives the real moderator and
reason. Lifts (unban, timeout cleared) revoke the matching active sanction, which
lets the case auto-close.

Requires the bot to have **View Audit Log** in the guild. Bot targets are
ignored.

| Audit action | Effect |
|---|---|
| `ban` | record `guild` / `ban` |
| `unban` | revoke active `guild` / `ban` |
| `kick` | record `guild` / `kick` |
| `member_update` (timeout set) | record `guild` / `mute` with `expires_at` |
| `member_update` (timeout cleared) | revoke active `guild` / `mute` |

## 6. Staff commands (`/mod case`)

| Command | Permission | Purpose |
|---|---|---|
| `create` | `case_create` | Open a case + first sanction (source-driven picker) |
| `view <reference>` | `case_view` | Sidebar fields + timeline (Linear-style) |
| `list <user\|guild>` | `case_list` | List a subject's cases |
| `sanction <reference>` | `case_sanction` | Add a sanction to a case |
| `revoke <reference>` | `case_sanction` | Revoke an active sanction |
| `edit <reference>` | `case_edit` | Edit the case reason |
| `note <reference> [internal]` | `case_note` | Add a public comment or internal note |
| `close <reference>` | `case_close` | Manually close / reopen (locks status) |

References are the public 6-char codes (e.g. `A7F2K9`). The `create` picker is
driven by the manual sources in the registry, so it scales automatically.

## 7. User-facing browsers (`/cases` & `/sanctions`)

Two commands expose the cases system to non-staff, both built on the single
reusable `CasesBrowserView` in `utils/cases_views.py` (Components V2 list +
detail screens, filters collected through a **Modal** — Modals V2):

| Command | Scope of the query | Who | Capabilities |
|---|---|---|---|
| `/cases` | subject = the caller (`discord_user`), any scope | anyone | **read-only**, all servers combined |
| `/sanctions` | scope = the current guild (`discord_guild`) | guild moderators | read **+ full case management** |

**List screen** — paginated overview (`PAGE_SIZE = 5`). A single *Filters*
button opens a modal to set: status (open/closed), sanction type, period
(24h / 7d / 30d / 90d), and a context filter (a server in `/cases`, a user in
`/sanctions`). A select opens any listed case ("zoom").

**Detail screen** — the full folder (fields, reason, sanctions, public
comments). Internal Moddy-staff notes (`event_type = note`) are **never** shown
here. In `/sanctions` the moderator gets every case action — add sanction,
revoke sanction, comment, edit reason, close/reopen — recorded with
`author/issuer = discord_user` (the moderator), never `moddy_staff`. `/cases`
shows no action buttons.

`/sanctions` is guild-only and requires a moderation permission in the guild
(ban / kick / timeout members, or manage server / administrator); only
guild-scoped cases are reachable, so guild moderators can never touch
`global`/`platform` (Moddy-team) cases.

Backing repository queries (`db/repositories/moderation.py`): `search_cases`
(filtered + paginated, carries `actions` and `has_active` per row),
`count_cases`, and `list_subject_scopes` (servers a subject has cases in, for
the `/cases` server filter).

## 8. Notes / open points

- **Deletion**: FKs use `ON DELETE CASCADE`; no command deletes cases (moderation
  history is preserved). Soft-delete can be added later if needed.
- **Multi-scope**: a case has a single scope. A single affair spanning scopes
  (e.g. network ban + platform ban) is modelled as linked cases sharing a
  `group_id`.
- **Evidence**: stored as `evidence` timeline events with a URL in `payload`.
